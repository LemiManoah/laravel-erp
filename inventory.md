# Inventory Module Notes

This document describes the inventory module as it exists now, after the move from the old `products` terminology to `inventory_items`.

It also captures the most likely additions if we keep expanding the module.

## Core Direction

The inventory design is now centered around four layers:

1. `inventory_items`
2. `inventory_item_prices`
3. `inventory_stocks`
4. `inventory_movements`

That means:

- inventory items are the master records
- default pricing lives outside the item table
- on-hand stock lives outside the item table
- the movement ledger is the source of stock history
- expiry and batch detail live on stock rows, not on the item master

## Inventory Item Layer

The `inventory_items` table is the master catalog.

It answers:

- what is this item
- what kind of item is it
- should it track stock
- can it be sold
- can it be purchased
- does it expire
- does it have variants
- what unit does it use

Current key fields:

- `item_category_id`
- `sku`
- `barcode`
- `item_type`
- `tracks_inventory`
- `is_sellable`
- `is_purchasable`
- `base_unit_id`
- `reorder_level`
- `has_variants`
- `parent_item_id`
- `has_expiry`
- `is_serialized`
- `name`
- `description`
- `is_active`

Removed from the old design:

- `allow_negative_stock`
- `reorder_quantity`

Those are intentionally gone now:

- negative stock is blocked by the inventory movement engine
- reorder quantity is not yet part of the replenishment workflow

## Item Categories And Types

We now support both item categories and item types.

Item categories:

- are tenant-defined
- help organize the catalog
- feed into SKU and barcode generation

Current item types:

- `service`
- `stock_item`
- `non_stock_item`
- `raw_material`
- `finished_good`
- `consumable`

The `InventoryItem` model includes scopes for working with these types:

- `ofItemType()`
- `services()`
- `stockItems()`
- `nonStockItems()`
- `rawMaterials()`
- `finishedGoods()`
- `consumables()`
- `stockTracked()`
- `sellable()`
- `purchasable()`
- `active()`

This gives us enough structure to support both simple trading businesses and businesses that distinguish between raw materials, finished goods, services, and consumables.

## SKU And Barcode Rules

SKU is no longer manually entered during item creation.

Current rule:

- SKU is autogenerated after the inventory item is saved

Current SKU shape:

- `ITI-{type-code}-{category-code}-{id}`

Example:

- `ITI-SI-FAB-0007`

Barcode is also system-managed.

Current rule:

- barcode is generated automatically only for sellable inventory items
- if an item is no longer sellable, the stored barcode is cleared

Current barcode shape:

- EAN-13 style numeric value derived from item type, item category, and record id

This keeps item setup simpler and avoids duplicate or inconsistent identifiers from manual entry.

## Pricing Layer

Default pricing lives in `inventory_item_prices`.

Current fields:

- `inventory_item_id`
- `sale_price`
- `purchase_price`

Current behavior:

- one default price row per inventory item
- the item exposes `sale_price` and `purchase_price` via accessors
- transactions can still override line pricing when needed

This is intentionally simple for now, while leaving room for future price lists or customer-specific pricing.

## Stock Layer

`inventory_stocks` holds the actual on-hand rows.

Key concepts:

- normal stock-tracked items typically have one stock row per item and location
- expiry-controlled items can have multiple stock rows per item and location
- batch and expiry details belong on stock rows, not on the master item

Important rules in the movement engine:

- non-expiry items use the null-batch stock row for the selected location
- expiry items require batch number and expiry date on stock-in
- expiry items require choosing a specific stock row on manual stock-out
- stock cannot go negative
- expired stock cannot be issued for sales

## Movement Ledger

`inventory_movements` is the ledger for stock changes.

The movement engine currently handles:

- opening stock
- purchase receipt
- sale issue
- sales return
- purchase return
- adjustment gain
- adjustment loss
- transfer out
- transfer in
- damage
- wastage
- expiry write-off
- harvest
- production output
- internal consumption

Important behavior:

- inventory transactions use row locking for consistency
- the engine updates the stock row and then writes the ledger row
- total quantity on hand is derived from stock rows, not stored directly on the item

## Current Operational Flows

### Inventory Item Setup

When creating an inventory item:

- the item master is saved
- SKU is generated automatically
- barcode is generated automatically if the item is sellable
- the default price row is stored separately
- no stock row is created automatically

### Inventory Receipts

Inventory receipts are now cart-style.

Current operator flow:

- choose one receipt header:
  - receipt type
  - stock location
  - movement date
  - overall notes
- add multiple line items in one receipt
- each line captures:
  - inventory item
  - quantity
  - unit cost
  - batch number if needed
  - expiry date if needed
  - line notes

Supported receipt-style movement types on the receipt page:

- opening stock
- purchase receipt
- sales return
- adjustment gain
- harvest
- production output

### Manual Inventory Movements

The generic inventory movement page still exists as the advanced fallback page.

That is useful when a business needs an operator-controlled ledger entry outside the more guided flows.

### Sales Issue

Inventory is issued from invoices, not from a separate POS screen yet.

Current sales behavior:

- issuing an invoice deducts inventory for stock-tracked items
- non-stock items and services do not produce stock movements
- expiry-controlled items use FEFO-style issuing:
  - earliest valid expiry first
  - expired rows are excluded
- cancelling an issued invoice restores inventory through return movements

### Purchasing

The purchasing flow is already present:

- purchase orders
- purchase receipts
- receipt against purchase order
- purchase returns

Current behavior:

- purchase orders record intended inbound quantities and costs
- purchase receipts post stock into the selected location
- purchase returns deduct stock from a specific stock row
- purchase receipts can update purchase order status to received

### Monitoring And Reporting

The inventory module currently includes:

- units of measure
- stock locations
- inventory stocks page
- inventory movements page
- inventory monitoring page
- inventory status report
- stock card report
- supplier purchasing report

Current monitoring/reporting capabilities include:

- low-stock visibility using `reorder_level`
- near-expiry stock
- expired stock
- stock by location
- per-item stock movement history
- supplier purchasing totals

## What Could Be Added Next

The biggest useful additions from here would be:

### 1. Over-The-Counter Sales / POS

Right now over-the-counter sales are handled through the invoice flow.

A dedicated POS page could combine:

- walk-in customer handling
- barcode search / quick item search
- cart entry
- issue inventory
- take payment immediately
- print receipt

### 2. Stock Counts / Stock Takes

This is one of the most valuable missing inventory operations.

Suggested additions:

- stock take sessions by location
- counted quantity capture
- variance calculation
- automatic adjustment posting after approval

### 3. Replenishment Suggestions

We currently have `reorder_level` but not a full replenishment workflow.

Possible additions:

- reorder suggestions by location
- suggested purchase quantities
- create purchase order from low-stock list

### 4. Reservations / Allocations

This matters once order volume grows.

Possible additions:

- reserve stock for confirmed sales orders
- show available-to-sell vs physical on-hand
- release reservations on cancellation

### 5. Stronger Serialized Item Support

The `is_serialized` flag exists, but the full workflow is not built out yet.

Possible additions:

- serial capture on receipt
- serial issue on sale
- serial traceability history

### 6. Variant-Aware Catalog Management

The model supports parent/variant relationships, but the operator workflow can be richer.

Possible additions:

- parent item with attribute-based variants
- variant matrix creation
- variant-specific barcode and pricing tools

### 7. Manufacturing / Conversion Flows

We already have movement types like `production_output` and `internal_consumption`.

Possible additions:

- bill of materials
- production orders
- raw material issue to production
- finished good output posting

### 8. Approval Workflows

Some businesses will want stronger controls around stock changes.

Possible additions:

- approval for adjustments
- approval for transfers
- approval for purchase returns

### 9. Better Costing

Unit cost is stored per stock row and movement, which is enough for the current stage.

Possible future additions:

- weighted average valuation
- FIFO valuation reporting
- inventory valuation report by location

### 10. Import / Bulk Maintenance

Useful for onboarding larger tenants.

Possible additions:

- bulk import of inventory items
- bulk opening stock import
- bulk price updates

## Summary

The inventory module is no longer product-centric in the old sense.

The current shape is:

- `inventory_items` for the master record
- `inventory_item_prices` for default prices
- `inventory_stocks` for on-hand rows
- `inventory_movements` for the ledger

And the main business rules now are:

- SKU is autogenerated
- barcode is autogenerated only for sellable items
- negative stock is blocked
- expiry-controlled stock uses batch-aware rows
- invoice issue handles stock deduction
- inventory receipts support multiple line items like a cart

That is a solid first-release inventory foundation, with the next highest-value additions being POS, stock takes, replenishment, reservations, and stronger serialized or manufacturing flows.
